# REWRITE IMPROVEMENTS

> 说明：本文档面向 GitHub 展示与面试讲解，重点说明“相较原作者，改写版做了哪些工程化改进”，以及这些改进背后的设计原理。
>
> 这里强调的是**重写与工程重构**，不是简单改几个变量名。所有核心模块都按新的结构重新组织，以便在不改变功能边界的前提下，获得更好的可维护性、可扩展性和部署能力。

## 1. 改写目标

这个改写版并不是单纯的“把原代码复制过来改一点”，而是围绕以下目标重新设计：

1. **保留原项目能力边界**
   - 继续提供健康检查、聊天、会话、检索、上传、LangGraph 入口等核心能力。
   - 保证对外接口路径尽量兼容，便于原有调用方迁移。

2. **重建内部实现结构**
   - 后端从单一骨架式组织，改成分层式结构。
   - 前端从零散页面式写法，改成页面层、功能层、组合式逻辑层、服务层的模块化结构。

3. **增加可部署性与可演示性**
   - 引入 Alembic 做数据库迁移。
   - 引入 Docker Compose 做一键编排。
   - 增加文档、截图、发布清单，适合 GitHub 展示。

4. **增强面试表达价值**
   - 不只是“能跑”，而是能解释：为什么这样拆分、为什么这样接 LLM、为什么这样做持久化。

## 2. 后端重写：从单文件思路到分层架构

### 2.1 原理概述

后端的核心变化是：把“HTTP 协议、业务逻辑、领域对象、基础设施”拆开。

这背后的原理是典型的**分层架构**思想：

- **API 层**：负责接收请求、校验参数、返回响应。
- **Service 层**：负责业务编排，把多个能力组合起来。
- **Domain 层**：负责表达业务对象、仓储接口等抽象。
- **Infra 层**：负责真正和数据库、外部 API、文件系统交互。
- **Core 层**：负责组装依赖，降低耦合。

这样做的好处是：

- 业务变化时，尽量只改一层。
- 单元测试更容易，因为业务逻辑不必直接依赖数据库。
- 后续接入 Redis、PostgreSQL、向量数据库、真实 LLM 时，只要替换基础设施实现即可。

### 2.2 代码上的体现

当前后端目录已经按这个方向组织：

- `backend_rewrite/app/api/`：路由层
- `backend_rewrite/app/services/`：业务编排层
- `backend_rewrite/app/domain/`：领域抽象
- `backend_rewrite/app/infra/`：SQLite、LLM 客户端等基础设施
- `backend_rewrite/app/core/`：容器装配与配置

入口在 `backend_rewrite/app/main.py`，通过 `create_app()` 构建应用，再由 `api_router` 挂载路由。

### 2.3 面试表达重点

可以这样表述：

> 我没有把后端写成一个“大而全的脚本”，而是按 API / Service / Domain / Infra 分层重写。这样做的核心价值是把变化点隔离开：接口协议变动影响 API 层，业务编排变动影响 Service 层，数据库或模型服务切换只影响 Infra 层，整体更适合后期维护和扩展。

## 3. 会话与消息：把“状态”从临时内存变成可持久化数据

### 3.1 原理概述

智能客服系统如果只靠内存保存会话，会有两个问题：

1. 服务重启后会话丢失；
2. 多用户、多会话时，无法支撑后续扩展。

所以改写版引入了**SQLite 持久化**，并配合 Alembic 管理结构演进。

这个设计的原理是：

- 会话和消息是业务状态，不应该只放在进程内存中。
- 数据必须可追踪、可恢复、可迁移。
- 用仓储模式把“业务逻辑”和“存储实现”解耦。

### 3.2 具体做法

当前实现中：

- 会话仓储默认使用 SQLite。
- 也保留 `memory` 回退选项，便于测试和演示。
- 数据库初始化由容器启动时自动执行。
- Alembic 负责迁移版本管理。

### 3.3 为什么不是直接写死 SQL

如果把 SQL 直接写在业务逻辑里：

- 业务代码会混入很多数据库细节。
- 后续改表结构会很痛苦。
- 单元测试也不容易隔离。

改成仓储模式后：

- Service 层只关心“我要保存会话”或“我要读消息”。
- SQLite、未来的 PostgreSQL、甚至远程数据库都能以相同接口替换。

### 3.4 面试表达重点

可以这样说：

> 这个项目把会话数据从内存状态升级成了 SQLite 持久化，并用 Alembic 做迁移管理。这样不仅解决了重启丢数据的问题，也让表结构能够随业务变化平滑演进，后续如果切到 PostgreSQL 也只需要替换基础设施层。

## 4. LLM 接入：从“固定模型”到“可切换适配层”

### 4.1 原理概述

大模型接入最容易踩的坑，是把某一家 API 写死在业务代码里。这样一来：

- 换模型成本高；
- 开发、测试、生产环境配置不一致；
- 流式和非流式接口容易写两套重复逻辑。

改写版采用了一个独立的 `LLMClient` 适配层，把模型能力封装成统一入口。

### 4.2 设计思想

适配层的核心原理是：**把外部服务差异收敛到一个边界层**。

当前支持的逻辑包括：

- `mock`：本地开发时快速联调，不依赖真实模型。
- `ollama`：本地模型服务。
- `deepseek` / `openai`：OpenAI 兼容接口。

这种设计的价值在于：

1. 业务层不关心“调用的是谁”；
2. 测试时可以快速切 mock；
3. 流式与非流式都走统一调用链路；
4. 后续扩展其他模型，只要继续加 provider 分支即可。

### 4.3 具体实现特点

当前 `backend_rewrite/app/infra/llm_client.py` 做了几件事：

- 统一解析 model 名称；
- 为不同 provider 组装不同请求格式；
- 把 API Key 放到请求头里；
- 支持 `complete()` 和 `stream()` 两种模式；
- 流式模式把上游流数据转换成下游可消费的 chunk。

### 4.4 为什么要支持“流式”和“非流式”两条链路

客服场景里，流式体验很重要：

- 用户能更快看到“正在回答”；
- 交互感更自然；
- 可以边出边看，降低等待焦虑。

但在某些场景下，非流式更适合：

- 需要完整答案再统一展示；
- 需要做后处理或审计；
- 某些环境只适合同步调用。

因此把两种模式都保留下来，是工程上更稳妥的做法。

### 4.5 面试表达重点

可以这样说：

> 我把 LLM 接入做成了 provider 适配层，而不是把某家模型的 API 写死在业务逻辑里。这样可以同时支持 mock、本地模型和 DeepSeek/OpenAI 兼容接口，流式和非流式复用同一套业务链路，后续切模型或换供应商的成本很低。

## 5. 前端重构：从“能展示”到“能组织业务能力”

### 5.1 原理概述

前端重写的重点不是 UI 变复杂，而是把界面从“单文件页面”升级成“模块化工作台”。

现在的组织方式是：

- `pages/`：页面组合层
- `features/`：具体业务模块
- `composables/`：交互状态与复用逻辑
- `services/`：HTTP 和业务 API 封装
- `shared/`：通用配置和类型

### 5.2 为什么要这样分

如果把所有逻辑都塞进一个 Vue 页面里，会出现：

- 页面越来越长；
- 状态越来越乱；
- 接口封装与 UI 混在一起；
- 后续新增模块很难维护。

模块化的原理是把不同关注点拆开：

- 页面负责“拼装”；
- 组件负责“展示”；
- composable 负责“状态变化”；
- service 负责“请求协议”；
- shared 负责“公共配置”。

### 5.3 当前页面结构

现在 `ChatWorkbenchPage.vue` 作为主页面，把：

- 健康检查卡片
- 会话面板
- 流式聊天卡片

组合到同一个工作台中。

这样做的意义是：

- 一眼能看出页面结构；
- 各功能模块之间边界清楚；
- 以后要加知识库、上传区、工具区时，可以继续按模块扩展。

### 5.4 面试表达重点

可以这样说：

> 前端不是单页堆逻辑，而是按页面层、功能层、组合式逻辑层和服务层拆分。这样能把 UI、状态和接口调用分离，页面后续扩展更稳定，也更方便做维护和演示。

## 6. 部署能力：从“本地能跑”到“一条命令起全栈”

### 6.1 原理概述

一个能展示到 GitHub 的项目，不能只停留在“我电脑上能跑”。

所以改写版加入了：

- 本地启动脚本；
- Docker Compose；
- Nginx 静态托管；
- 数据卷挂载；
- 环境变量配置。

这背后的原理是：**把运行环境和代码分离**。

### 6.2 Docker Compose 的意义

Compose 的价值在于：

- 一个命令启动前后端；
- 前端镜像使用 Nginx 做静态托管；
- 后端镜像负责 API 服务；
- 数据持久化通过卷挂载完成。

对于演示和部署来说，这比让别人手动装依赖更可靠。

### 6.3 Nginx 为什么适合前端静态托管

前端构建后本质上是一组静态资源。Nginx 的优势：

- 性能好；
- 配置简单；
- 适合做反向代理；
- 便于把 `/api/*` 转发给后端。

这样前后端可以在浏览器里看起来像同一个站点，部署也更清晰。

### 6.4 Alembic 为什么要加入

数据库结构会变化：

- 新增字段；
- 调整表结构；
- 扩展多店铺字段；
- 扩展用户权限体系。

Alembic 的作用就是：

- 记录结构版本；
- 支持升级和回滚；
- 让团队协作时更可控。

### 6.5 面试表达重点

可以这样说：

> 我补了 Docker Compose、Nginx 和 Alembic，让项目从“开发机可运行”升级到“可交付、可部署、可迁移”的状态。这样不仅便于面试展示，也方便后续发到 GitHub 给别人直接复现。

## 7. 合规原创改版：为什么强调“重写”而不是“抄改”

### 7.1 原理与边界

这个项目对外的定位是“合规原创改版”，核心原则是：

- 保留功能边界；
- 重新组织代码；
- 不复制原作者实现细节；
- 对来源、许可证、第三方依赖做说明。

### 7.2 为什么要这样做

这样做有三个现实价值：

1. **面试可解释**：你可以讲清楚每个模块为什么这么设计。
2. **GitHub 可展示**：仓库看起来是一个完整工程，而不是零散改动。
3. **风险更可控**：文档和证据链更完整，便于发布前自查。

### 7.3 建议的表述方式

建议使用：

- “重写版”
- “改版实现”
- “分层重构”
- “接口兼容”
- “工程化增强”

尽量避免绝对化措辞，例如：

- “完全替代”
- “百分百准确”
- “业内最强”

## 8. 面向淘宝商家的场景优化

### 8.1 为什么要加场景化说明

淘宝商家客服和通用聊天机器人不一样：

- 更关注售后规则；
- 更关注物流和时效；
- 更关注价格、赔付和合规表述；
- 更容易涉及高风险承诺。

所以项目里专门补了 `docs/DEPLOYMENT_TAOBAO_SELLER_NOTES.md`。

### 8.2 设计原则

- 业务规则优先于自由生成；
- 敏感问题尽量模板化或人工接管；
- 店铺数据应尽量隔离；
- 必须控制日志与隐私数据暴露。

### 8.3 面试表达重点

可以这样说：

> 这个项目不是泛聊天机器人，而是明确面向淘宝商家客服场景做过约束设计。比如售后规则、敏感话术、店铺隔离和日志脱敏，这些都是业务落地时必须考虑的。
